## 2. 이 해커톤의 본질

**"2시간 + 1인 + 주제 직전 공개"**라는 세 가지 제약이 모든 전략을 결정한다.

- 팀이 없으니 **분업 전략 = 무효**
- 시간이 짧으니 **풀스택/인증/결제 = 거의 불가능**
- 평가 기준이 "**날카로움 + 완성도**"이므로 **크고 두루뭉술한 것 < 작고 또렷한 것**

→ **"좁고 또렷한 한 가지"를 2시간 안에 배포된 URL로 보여주는 게임.**

## 3. 사전 준비 (D-day 전이 승부의 70%)

### 3.1 코드 보일러플레이트

- `git clone` 후 30초 안에 `npm run dev` 가 뜨는 상태로 만들어둔다.
- 이 레포에 세팅된 스택:
  - **frontend/**: Vite + Svelte + TypeScript + SCSS(BEM) + Supabase 클라이언트
  - **backend/**: Node.js + TypeScript (필요 시 NestJS)
  - 모노레포 도구 없이 폴더 분리 + 루트 `package.json`으로 스크립트 프록시

### 3.2 AI 코딩 환경

- Claude Code, Cursor, v0.dev, Bolt.new, Lovable 중 **본인 손에 가장 잘 붙는 1~2개 마스터**
- 자주 쓰는 프롬프트 패턴, 단축키 손에 익혀두기
- Claude Code는 이 레포에서 바로 작업 가능

### 3.3 API 키 미리 발급

| 서비스 | 용도 | 비고 |
|---|---|---|
| Supabase | DB / Auth / Storage | 무료 티어, 프로젝트 미리 생성 |
| Anthropic / OpenAI | LLM | 필요 시. 결제 등록까지 끝내두기 |
| Vercel | 배포 | GitHub 연동 미리 |
| (선택) Clerk | Auth | Supabase Auth면 충분 |
| (선택) Stripe | 결제 | 거의 안 씀 |

> 당일 가입은 절대 금물. 메일 인증 한 번이면 5분 날아감.

### 3.4 배포 파이프라인 검증

- D-day 전에 **반드시 한 번 Vercel에 배포해서 URL 확인**
- 도메인까지 살아있는 상태로 대기 (예: `indie-hack.vercel.app`)
- **데모는 로컬 서버가 아닌 URL로** 보여줘야 임팩트 큼
- QR 코드도 미리 생성해서 슬라이드에 박아둘 수 있게

### 3.5 UI 스니펫

- 자주 쓰는 컴포넌트 (로그인, 카드, 채팅 UI, 결과 페이지) 스니펫화
- 30초 안에 복붙 가능하게

## 5. 아이디어 선정 원칙 (1등을 가르는 부분)

평가 기준에 **"아이디어의 날카로움"**이 들어있다. 이게 결정적이다.

### 5.1 좋은 아이디어의 4가지 조건

1. **타겟 1명**으로 좁혀라
   - ❌ "개발자를 위한 협업 도구"
   - ✅ "오늘 회의록 정리에 30분 쓰는 PM 김민준을 위한 1분 요약기"
2. **30초 안에 가치가 보여야**
   - 데모 본 순간 "오 이거 나 쓰겠다"가 나와야 함
3. **2시간에 만들 수 있어야**
   - "AI가 잘하는 것" ∩ "1인 + 2시간"의 교집합
4. **데모 가능한 결과물**
   - 콘솔 로그가 아니라 "보이는 결과"

### 5.2 금지 카테고리 (시간 못 맞춤)

- 로그인/회원가입 시스템
- 결제
- 복잡한 DB 스키마 (테이블 3개 이상)
- 모바일 네이티브 앱
- 실시간 동기화 / 멀티유저
- 무거운 ML 학습

### 5.3 추천 형태

- **단일 페이지 웹앱** + API 호출 1~2개 + 결과 시각화
- 입력 → 변환/요약/추천/구조화 → 결과
- "AI가 ~~ 해주는 가장 빠른 방법" 패턴

## 6. 완성도를 만드는 디테일

발표 때 심사위원 눈에 박히는 차이는 디테일에서 나온다.

- **빈 상태(empty state) 절대 금지** → 시드 데이터 미리 넣고 시작
- **로딩 / 에러 처리** → 데모 중 한 번이라도 멈추면 끝. AI에게 "skeleton loader 추가" 한 줄
- **랜딩 페이지 한 줄 카피** → 제품의 가치를 한 문장으로
  - 형식: "X를 Y로 바꿔주는 가장 빠른 방법"
- **배포 URL + QR 코드** → 슬라이드 마지막에. 심사위원이 그 자리에서 만져볼 수 있게
- **파비콘 + OG 이미지** → 별 거 아닌 거 같지만 "완성된 것" 같은 인상에 큰 차이
- **다크모드 1줄** → Tailwind 기본으로 가능. 보기 좋음
- **모바일 대응** → 심사위원이 폰으로 QR 찍어서 들어감. 깨지면 안 됨

## 7. 엘리베이터 스피치 (16:30~17:30)

### 7.1 1분 공식

```
1. 문제 (15초) : "○○하는 사람들은 □□할 때마다 △△로 고통받습니다."
2. 솔루션 (15초): "그래서 [제품명]을 만들었습니다. ◇◇하면 ◎◎해줍니다."
3. 데모 (20초) : 실제 화면 1번 시연 (말하지 말고 보여주기)
4. 마무리 (10초): "지금 [URL]에서 써보실 수 있습니다."
```

### 7.2 발표 원칙

- **데모 7 : 말 3** 비율
- "제품 자랑"이 아니라 **"한 사람의 고통 → 해결"의 스토리**
- 로컬 데모 X → 배포된 URL을 큰 화면에
- 발표 시작 전 노트북 화면 100% 밝기 + 다른 탭 다 닫기
- 발표 중 새로고침 / 인터넷 끊김 대비해서 **백업 영상** 1개 미리 녹화

### 7.3 슬라이드 구성 (있다면)

1. 제품명 + 한 줄 카피
2. 누구의 어떤 고통인가
3. 어떻게 해결하는가 (스크린샷 1장)
4. 데모 (실시간)
5. URL + QR

## 9. 1등을 만드는 마지막 팁

- **혼자 하는 해커톤**이라 옆에서 도와줄 사람이 없다. **AI가 곧 팀원**. 말 거는 속도가 곧 개발 속도.
- 인디 크루 같은 곳은 **"말 잘하는 사람"보다 "진짜 동작하는 거"**를 본다.
- 완벽한 거 1개가 부족한 거 3개를 이긴다. **자르고 또 잘라라**.
- 데모 중 멈추면 1등은 못 한다. **빈 상태 / 로딩 / 에러 처리 = 무조건**.
- 발표는 **30초 안에 가치가 보여야** 한다. 그 30초 시나리오를 미리 만들어둬라.
